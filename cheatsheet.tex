\section{Outcomes}

\begin{tabular}{lll}
    PASSED  & \Large\textcolor{termgreen}{\texttt{.}} & All assertions passed, no exceptions   \\[.5em]
    FAILED  & \textcolor{termred}{\texttt{F}}         & Assertion failed or exception occurred \\
    ERROR   & \textcolor{termred}{\texttt{E}}         & Exception occurred outside of test     \\
            &                                         & (e.g.\ in a \emph{fixture})            \\[.5em]
    SKIPPED & \textcolor{termyellow}{\texttt{s}}      & The test was skipped                   \\
            &                                         & (e.g.\ via \mono{@pytest.mark.skipif}) \\
    XFAIL   & \textcolor{termyellow}{\texttt{x}}      & An expected failure occurred           \\
    XPASS   & \textcolor{termyellow}{\texttt{X}}      & An unexpected success occurred         \\
            &                                         & (expected to fail but passed)
\end{tabular}

\vspace{.5em}
pytest 9 also introduces subtests with SUBPASSED (\textcolor{termgreen}{\texttt{u}}), SUBFAILED (\textcolor{termred}{\texttt{u}}), SUBSKIPPED (\textcolor{termyellow}{\texttt{-}}) \& SUBXFAIL (\textcolor{termyellow}{\texttt{y}}).
Only SUBFAILED is visible by default, rest needs \mono{-v}.

\section{Strict mode}
pytest 9: \ini{strict = true} enables: \vspace{-.5em}

\begin{itemize}[itemsep=0pt]
    \item \inikey{strict_config} (error on unknown options)
    \item \inikey{strict_markers} (error on unknown markers)
    \item \inikey{strict_parametrization_ids} \\ (error on duplicate test IDs)
    \item \inikey{strict_xfail} (error on unexpected passes)

Older pytest versions: Set \ini{xfail_strict = true} and \ini{addopts = --strict-config --strict-markers}

\linksection{approx}{pytest.approx}

Floating point comparisons need a tolerance:
\vspace{-.6em}

\begin{minted}[autogobble, highlightlines=2]{python}
    res = add(0.2, 0.1)
    assert res == pytest.approx(0.3)
\end{minted}
\vspace{-.6em}

Can override rel./abs.\ tolerance, e.g.\ 20 \textpm{} 2\textdegree: \\
\py[escapeinside=||]{assert temperature == pytest.approx(20, abs=2)}
default: 1e-6 relative / 1e-12 absolute

Supports various data types: Sequences of numbers \\ (e.g.\ lists or tuples), dictionary values, \mono{numpy} arrays

\linksection{skipping}{Skip and xfail}
\textbf{Skip a test if}:
\begin{itemize}[itemsep=0pt]
    \item It cannot run at all on a certain platform
    \item It cannot run because a dependency is missing
\end{itemize}

\vspace{-.5em}
$\Rightarrow$ Test function is not run, result is \enquote{skipped} (\textcolor{termyellow}{\texttt{s}})

Use \py{@pytest.mark.skip} (instead of \mono{skipif}) \\ for unconditional skipping.
\vspace{-.5em}

\begin{minted}[autogobble,escapeinside=||]{python}
    @pytest.mark.skipif(
        # condition
        sys.platform == "win32",
        # text shown with -v
        reason="Linux only",
    )
    def test_linux():
        |\codeskip|
\end{minted}
\vspace{-.5em}

Skip from inside a test or fixture function by calling \py{pytest.skip(reason)}.
\vspace{1em}

\textbf{\enquote{xfail} (\enquote{expected to fail}) a test if}:
\begin{itemize}[itemsep=0pt]
    \item The implementation is currently lacking
    \item It fails on a certain platform but should work
\end{itemize}

\vspace{-.5em}
$\Rightarrow$ Test function is run, but result is \enquote{xfailed} (\textcolor{termyellow}{\texttt{x}}), \\ \phantom{$\Rightarrow$} instead of failed (\textcolor{termred}{\texttt{F}}). Unexpected pass: XPASS (\textcolor{termyellow}{\texttt{X}}).

\begin{minted}[autogobble,escapeinside=||]{python}
    @pytest.mark.xfail(
        # condition optional
        reason="see #1234",
    )
    def test_new_api():
        |\codeskip|
\end{minted}

Use \mono{pytest.raises} to test a \enquote{bad case} but \textsb{intended behavior}. Use \mono{pytest.mark.xfail} to mark a test where the implementation behaves in \textsb{unintended} ways: The test \textsb{should work}, but currently does not (e.g.\ upstream bug).

\linksection{parametrize}{Parametrization}

\begin{minted}[autogobble]{python}
    @pytest.mark.parametrize("a, b, expected", [
        (1, 2, 3),
        (2, 3, 5),
    ])
    def test_add(a, b, expected):
        assert add(a, b) == expected
\end{minted}

\linksection{mathspp-param}{Permutations}
\begin{minted}[autogobble]{python}
    @pytest.mark.parametrize("a", [1, 2])
    @pytest.mark.parametrize("b", [3, 4])
    def test_permutations(a, b):
        assert add(a, b) == a + b
\end{minted}

Test runs 4 times: 1 + 3 \quad 1 + 4 \quad 2 + 3 \quad 2 + 4.

Can be useful for bool flags (e.g. \py{strict=True/False} where all cases should behave the same).

\linksection{param}{pytest.param}

Instead of tuples, \py{pytest.param} objects can be passed to \mono{parametrize}, which additionally take \mono{marks} and \mono{id}:

\begin{minted}[highlightlines=2-6]{python}
@pytest.mark.parametrize("a, b, op, res", [
    pytest.param(
        2, 3, "**", 8,
        marks=pytest.mark.xfail(reason="..."),
        id="pow",
    ),
    pytest.param(1, 2, "+", 3, id="add"),
    pytest.param(3, 1, "-", 2, id="sub"),
])
def test_calc(a, b, op, res):
    assert calc(a, b, op) == res
\end{minted}

\begin{tabular}{ll}
    \textsb{Before:}                                             & \textsb{After:}                                                                            \\
    \mono{test_calc[2-3-**-8]} (\textcolor{termred}{\texttt{F}}) & \monot{test\_calc[\highlightbox{pow}]} \highlightbox{(\textcolor{termyellow}{\texttt{x}})} \\[.5em]
    \mono{test_calc[1-2-+-3]}                                    & \monot{test\_calc[\highlightbox{add}]}                                                     \\[.3em]
    \mono{test_calc[3-1---2]}                                    & \monot{test\_calc[\highlightbox{sub}]}
\end{tabular}

\linksection{fixtures}{Fixtures}

\end{minted}

\linksection{fixt-override}{Fixture visibility}

Fixtures defined\ldots{}
\begin{itemize}[itemsep=0pt]
    \item \ldots{}as methods of a test class are available only to test
          methods on that class.
    \item \ldots{}in a test module are available only to tests in that
          module.
    \item \ldots{}in a \mono{conftest.py} file are available to tests in
          that directory and subdirectories.
    \item \ldots{}in a plugin (or built into pytest) are available everywhere.
\end{itemize}

Fixtures in a more specific location shadow those in a more general location.

\linksection{fixt-yield}{Cleanup in fixtures}
A fixture can use \py{yield} instead of \py{return}
to run cleanup steps after the test. Everything before the \py{yield} is setup,
at the \py{yield} the test function runs, everything after is teardown:

\begin{minted}[autogobble,highlightlines={5,8,9}]{python}
    @pytest.fixture
    def connected_client() -> Iterator[Client]:
        client = Client()
        client.connect()
        yield client
        client.disconnect()

    def test_client(connected_client: Client):
        print("in the test")
\end{minted}

\textbf{Beware:}

    \begin{description}[leftmargin=!, labelwidth=.7em, itemsep=0pt]
        \item [+] Faster tests (4s $\rightarrow$ 2s)
        \item [--] Less isolation between tests, \mono{test_b} fails:
    \end{description}
\end{minipage}
\hspace{2mm}
\begin{minipage}[t]{.45\linewidth}

        >   assert not rpn.stack
        E   assert not [42]
    \end{minted}
    % VS Code workaround
    \if 0
        )
        \end{minted}\fi
\end{minipage}
\vspace{0.5em}

\textbf{How to avoid this pitfall?}

\begin{itemize}[itemsep=0pt]
    \item Can you make the return value immutable somehow?
    \item Can you copy it (e.g. \mono{deepcopy})?
    \item Can you reset the state between tests with a second fixture? Or maybe even take a snapshot before the test and restore it after?
\end{itemize}

\section{Combining fixture scopes}

Using two fixtures, you can combine the benefits of caching and isolation:

\begin{minted}[autogobble, escapeinside=||,highlightlines={1,8}]{python}
@pytest.fixture(scope="module")
def rpn_instance() -> RPNCalc:
    time.sleep(2)
    return RPNCalc(Config())

@pytest.fixture
def rpn(rpn_instance: RPNCalc):
    rpn_instance.stack.clear()
    return rpn_instance

def test_a(rpn: RPNCalc):
    rpn.stack.append(42)
    assert rpn.stack == [42]

def test_b(rpn: RPNCalc):
    assert not rpn.stack
\end{minted}

\mono{rpn_instance} uses caching and does some sort of slow operation
(initializing database, starting external process, initializing a device, etc.),
but it's not used directly in tests. Instead, \mono{rpn} is used, which in some
way resets the state of the underlying object before every test.

        \draw [-] (instance) -- (rpnval);

\linksection{capturing}{Capturing fixtures}
\begin{description}
    \item[{\mono{capsys}}~] Capturing stdout/stderr in a test by overriding \py{sys.stdout},
          e.g.\ for \py[escapeinside=||]{print(|\codeskip*|)}.
    \item[\mono{capfd}~] \textsb{Capturing stdout/stderr in a test at file descriptor level,}
          e.g.\ for subprocesses or libraries printing from C code. \\[.5em]
          Type for both: \monot{pytest.CaptureFixture[\textcolor{pygkw}{str}]}
    \item[{\parbox[t]{6.5em}{\mono{capsysbinary}/\\\mono{capfdbinary}}}]
          \parbox[t]{6cm}{
              Like \mono{capsys}/\mono{capfd}, \\ but for binary data. \\
              Type: \monot{pytest.CaptureFixture[\textcolor{pygkw}{bytes}]}
          }
    \item[\mono{caplog}~] \textsb{Capturing logging output from a test.} \\
          Type: \mono{pytest.LogCaptureFixture}
\end{description}
\vspace{-.5em}

\mono{capsys} vs. \mono{capfd}: If in doubt, use \mono{capfd} to ensure everything is captured
(built-in pytest capturing also uses \mono{fd} by default).

\begin{minted}[autogobble, escapeinside=||]{python}
    def test_capfd(capfd):
        subprocess.run(["ls"])
        out, err = capfd.readouterr()
        assert out == "..."
\end{minted}

\vspace{-.5em}
Every process has \emph{two} outputs: \mono{stdout} and \mono[termred]{stderr}:
\vspace{-.5em}

\begin{minted}[autogobble,escapeinside=||]{text}
    $ ls doesnotexist /home
    |\textcolor{termred}{ls: cannot access 'doesnotexist': \codeskip}|
    |\textcolor{codeblue}{/home:}|
    |\textcolor{codeblue}{username}|
\end{minted}

\begin{minted}[autogobble,escapeinside=||]{text}
    $ ls doesnotexist /home |\highlightbox{> /dev/null}|
    |\textcolor{termred}{ls: cannot access 'doesnotexist': \codeskip}|
\end{minted}
\vspace{-.5em}

We get them separately with \mono{capfd.readouterr()}:

\begin{tabular}{l>{\columncolor{highlightcolor}}l>{\columncolor{highlightcolorred}}l}
                                       & stdout       & \highlightbox[highlightcolorred]{stderr} \\[.75em]
    \py{print("out 1")}                & \mono{out 1} &                                          \\
    \py{print("out 2")}                & \mono{out 2} &                                          \\
    \py{print("err", file=sys.stderr)} &              & \mono{err}                               \\
\end{tabular}

\vspace{-.25em}
\hspace{.2em} \py{out, err = capfd.readouterr()}  \\[.25em]
\hspace{1em} $\Rightarrow$ \hspace{.5em} \highlightbox{\mono{out}}: \py{"out 1\nout2\n"} \\[-.13em]
\hspace{1em} \phantom{$\Rightarrow$} \hspace{.5em} \highlightbox[highlightcolorred]{\monot{err\vphantom{t}}}: \py{"err\n"} \\[1em]

\mono{caplog} lets you access logging records. The \mono{.records} attribute provides full records which can be inconvenient (timestamps, etc.),
convenience attributes are available:

\vspace{-.5em}
\begin{minted}[autogobble,highlightlines={3,7-10}]{python}
def test_msgs(caplog: pytest.LogCaptureFixture):
    logging.warning("some msg")
    assert caplog.messages == ["some msg"]

def test_tpl(caplog: pytest.LogCaptureFixture):
    logging.warning("some msg")
    assert caplog.record_tuples == [
        # logger, loglevel,       message
        ("root", logging.WARNING, "some msg"),
    ]
\end{minted}

\linksection{monkeypatch}{Monkeypatching}

The \mono{monkeypatch} fixture allows to temporarily change state for the
\textbf{duration of a test function execution}:

\begin{itemize}[itemsep=0pt]
    \item Modify attributes on objects, classes or modules \\ (\mono{setattr}, \mono{delattr}). \\
          This also works with functions/methods, as those are attributes of the respective module/class/instance.
    \item Modify environment variables (\mono{setenv}, \mono{delenv})
    \item Change current directory (\mono{chdir})
    \item Modify dictionaries (\mono{setitem}, \mono{delitem})
    \item Adjust \mono{sys.path} for importing (\mono{syspath_prepend})
\end{itemize}

\begin{minted}[autogobble,escapeinside=||,highlightlines=7-12]{python}
def print_info():
    path = os.environ.get("PATH", "")
    print(f"platform: {sys.platform}")
    print(f"PATH: {path}")

def test_a(monkeypatch: pytest.MonkeyPatch):
    monkeypatch.setattr(
        sys,         # where to patch
        "platform",  # attribute name (str!)
        "MonkeyOS",  # new value
    )
    monkeypatch.setenv("PATH", "/zoo")
    print_info()
    assert False

def test_b():
    print_info()
    assert False
\end{minted}
